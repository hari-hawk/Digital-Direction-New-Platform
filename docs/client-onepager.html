Automated Inventory Extraction

From raw carrier documents to a standardized 60-field service inventory — POC overview
Prepared for Digital Direction
April 2026

What it does. Takes a folder of mixed carrier documents — invoices, CSRs, contracts, service reports, emails — and produces a clean, standardized 60-field service inventory that feeds directly into audit, optimization, and invoice workflows. Every value is traceable back to the source document, page, and section it came from.

StageWhat happensWhy it matters
1. Classify Detects carrier, document type, and format variant via filename patterns, first-page text, and an LLM fallback. Extracts account numbers up front. Same carrier ships multiple formats (AT&T has box-format and section-marker CSRs). Routing decisions happen here so the right parser and prompt are used.
2. Parse Three paths chosen per document: Docling for visual layouts, pdfplumber for mainframe CSRs, pandas for CSV/XLSX. Large bills are chunked at sub-account boundaries. A 472-page Windstream bill becomes ~90 focused chunks instead of one unreadable blob. Two-column AT&T layouts are read in the correct order.
3. Extract Per-chunk LLM extraction (Gemini 2.5) into the 60-field schema, with carrier-specific domain knowledge (USOC codes, field codes, line types) injected from YAML configs. Runs in parallel. A 472-page bill extracts in ~10–15 seconds, not 15 minutes. Only extracts what's explicit in the document — no hallucinated values. Every field carries a confidence score.
4. Merge Cross-references every document for the same account: invoice for MRC/quantity, CSR for phone numbers and USOCs, contract for term/dates. Rule-based first, LLM (Claude) only for conflicts and gaps. No single document has all 60 fields. This stage fills the gaps and resolves conflicts using priority rules you can audit.
5. Validate Regex checks (account #, phone, zip), cross-field math (qty × unit cost = MRC), summary-total reconciliation, and per-field confidence scoring. Catches extraction errors before they reach your reviewers.

Output — Every field is auditable

Each row is tagged as a Service (S) summary or Component (C) line item, and every field carries:

  • Value — extracted from the document
  • Source — document ID, page, section
  • Confidence — routed for review accordingly

Human-in-the-Loop & Self-Healing

  • Review UI shows the source document alongside extracted fields; reviewers correct LOW/MEDIUM, bulk-approve HIGH.
  • Corrections feed three loops: few-shot examples for future extractions, YAML domain-knowledge updates, adaptive confidence thresholds.
  • Unknown formats are flagged, never silently failed — a new carrier or format variant is resolved by a YAML config, not a code change.

Why this holds up: Carrier logic lives in YAML, not code — adding a new carrier is a config change, not an engineering project. Every field is auditable to source. Partial document sets still produce useful output, with missing fields marked MISSING (not guessed) and filled in when new documents arrive. Rule-based merge before LLM merge keeps cross-reference cost 60–80% lower than a naive approach.
